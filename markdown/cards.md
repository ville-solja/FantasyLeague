# Cards — Current Implementation

This document describes how cards are created, drawn, and scored as the system currently works.
It is written against the code, not the intended design — gaps between the two are noted explicitly.

---

## Card Data Model

```
cards
  id          integer  primary key
  player_id   integer  → players.id
  owner_id    integer  → users.id  (NULL = unclaimed, still in deck)
  card_type   string   "common" | "rare" | "epic" | "legendary"
  league_id   integer  → leagues.id
  is_active   boolean  true = in upcoming week's active roster
```

Cards have no stats, no modifier fields, and no quality multiplier column. Rarity (`card_type`) is stored but has no effect on scoring.

---

## Card Creation

Cards are generated by `seed_cards(league_id)` in `backend/seed.py`, called automatically after each league ingestion.

**Source of players:** any player who has at least one entry in `player_match_stats` for the given league. Players with no match data are not included.

**Cards generated per player:**

| Rarity    | Qty | In-game colour |
|-----------|-----|----------------|
| legendary | 1   | golden         |
| epic      | 2   | purple         |
| rare      | 4   | blue           |
| common    | 8   | white          |

**Total:** 15 cards per player.

The function is idempotent — if any card already exists for a given `(player_id, league_id)` pair the whole player is skipped.

---

## The Deck

The deck is the pool of unclaimed cards (`owner_id IS NULL`). `GET /deck` returns a count of unclaimed cards per rarity:

```json
{ "common": 320, "rare": 160, "epic": 80, "legendary": 40 }
```

All rarities share the same pool — there are no separate draws per rarity.

---

## Drawing a Card

`POST /draw` with `{ user_id }`.

**Guard:** a user can draw only if they have at least 1 token (`user.tokens > 0`). Each draw decrements `tokens` by 1. New users receive `INITIAL_TOKENS` tokens on registration (default 5). Admins can grant tokens via `POST /grant-tokens`.

**Selection:** all unclaimed cards are fetched into memory and one is chosen with `random.choice()` — uniform random across the entire unclaimed pool. No weighted probability logic exists. The rarity distribution of draws is therefore determined entirely by how many of each rarity remain unclaimed. Early in a season (full deck) the approximate odds are:

| Rarity    | Chance |
|-----------|--------|
| common    | 53%    |
| rare      | 27%    |
| epic      | 13%    |
| legendary | 7%     |

These shift as cards are claimed.

**After draw:**
- `owner_id` set to the drawing user
- If the user has fewer than 5 active cards → `is_active = true` (auto-fills roster)
- Otherwise → `is_active = false` (goes to bench)

**Return value:** card id, card_type, player_name, avatar_url, team_name, is_active, tokens (updated balance).

> ⚠️ No duplicate-player guard exists. A user can draw multiple cards for the same player across any rarity.

## Roster Management

`POST /roster/{card_id}/activate` — moves a benched card into the active roster (max 5 cards).
`POST /roster/{card_id}/deactivate` — moves an active card to bench.

These operations are only available for the upcoming (unlocked) week. Once a week's start time passes the roster is locked and a `WeeklyRosterEntry` snapshot is written — see `markdown/plan-weekly-roster-locking.md`.

## Scoring

`backend/scoring.py` contains the entire scoring function:

```python
def fantasy_score(p, weights):
    return sum(weights.get(key, 0) * p.get(key, 0) for key in weights)
```

A weighted sum of raw match stats. Weights are stored in the `weights` table and configurable by admins.

**Default weights:**

| Stat         | Default weight | Meaning                  |
|--------------|---------------|--------------------------|
| kills        | 3.0           | Kills in match           |
| assists      | 2.0           | Assists in match         |
| deaths       | −1.0          | Deaths (penalty)         |
| gold_per_min | 0.02          | GPM                      |
| obs_placed   | 1.0           | Observer wards placed    |
| sen_placed   | 1.5           | Sentry wards placed      |
| tower_damage | 0.002         | Tower damage dealt       |

**Rarity has no effect on scoring.** A legendary and a common card for the same player earn exactly the same points. Rarity is a hook for the future modifier system, not a current scoring factor.

## Modifiers

Not yet implemented. The `Card` model has no modifier columns and scoring applies no per-card adjustments.

The design intent (from `markdown/plan-traits.md`) is a tiered trait system:

| Rarity    | Traits assigned |
|-----------|----------------|
| common    | 1 × Tier 1     |
| rare      | 1 × Tier 1–2   |
| epic      | 2 traits       |
| legendary | 2–3 traits     |

Tier 1 traits are simple weight multipliers (e.g. +20% kills weight). Higher tiers add conditional bonuses or trade-offs. See `plan-traits.md` for the full design.
